# v0.2 Agent Loop & Harness Plan

> 这份文档是 v0.2 的实施参考，由 2026-04-23~24 多轮讨论整理。
> 已基于 pi-coding-agent spike 结果（2026-04-24, `docs/spike-pi-results.md`）更新。
> Spike 结论：✅ pi-coding-agent 满足全部 v0.2 硬需求，session / bash / capability / provider 全交给 pi。

## 0. 总原则

**默认抄 coding agent 约定，只在 design-specific 场景做针对性扩展**。

具体来说：
- pi-coding-agent / Claude Code / Cursor 有共识的做法 → 直接抄
- 三家不一致 → 挑最简单的
- 三家都没有，我们真有独特需求 → 才讨论发明

YAGNI 原则：早期 v0.x 可随意 breaking change，v1.0 才承诺向前兼容。

## 2. 核心架构

### 2.1 存储模型：去 SQL，走 JSONL + filesystem

```
~/Library/Application Support/open-codesign/
  config.toml              # BYOK 配置（provider / API key）
  sessions/
    <session-uuid>.jsonl   # 每个 session（= design）一个 JSONL
  thumbnails/              # 缩略图 cache
  settings.json            # 全局 allowedTools + 偏好
  allowed-commands.json    # 全局 bash allowlist（也可以合进 settings.json）

<user-workspace>/          # 用户的真 workspace（由 Settings 默认根配置）
  (用户文件)
  AGENTS.md                # 用户写的项目指令（可选）
  DESIGN.md                # 用户写的品牌 token（可选，Google spec）
  .codesign/
    settings.json          # per-design allowlist + 偏好
    settings.local.json    # 个人偏好（默认 gitignore）
```

**砍掉的 SQLite 内容**：

- `designs` 表 → session JSONL header 的 `cwd` + `name` 字段
- `design_files` 表 → workspace 磁盘（真源）
- `snapshots` / `snapshot_files` → 不做版本引擎（v0.3+）
- `chat_messages` → session JSONL message entries
- `comments` → session JSONL 里带 anchor 的 user message
- `diagnostic_events` → 可选 append-only log 文件

### 2.2 Design == Session

没有"project"概念。Design 就是 pi session，一个 JSONL 文件。

- session header 格式 100% 用 pi v2 原生，**不加字段**（包括不加 designId）
- `cwd` 字段 = workspace 路径
- `name` 字段 = 用户给的 design 名（pi 的 `/name` 命令）
- 多个 session 可共 cwd（= "对同一 codebase 的多个对话"）
- Sidebar 扁平展示 session，按 `lastOpenedAt` 排
- Fork = 新 session（pi 的 `/fork`，我们引擎层支持但 UI 暂不暴露）

### 2.3 Workspace 统一模型

- 每个 design **必有** workspace（无 sealed mode）
- Settings 里一个配置：**默认 workspace 根目录**（默认 `~/Documents/CoDesign/`）
- New Design 时默认路径 `<默认根>/<design名>/`，用户可自定义
- PR #173 的 Skip 按钮去掉

## 3. 工具清单（15 个，spike 后修订）

**总计 15 个工具暴露给 agent**：pi-coding-agent 自带 7 个 built-in（执行层我们不写，只 hook 拦截）+ 我们写 8 个（设计 domain 6 + 生命周期 2）。

### Pi 自带 7（spike 验证白送）

| 名称 | 我们做啥 |
|---|---|
| `read` | 直接用，pi 自带类似 view_budget 机制 |
| `write` | 直接用 |
| `edit` | str_replace 风格 |
| `bash` | 直接用，**通过 `tool_call` hook 拦截做权限**，不写新 bash 工具 |
| `grep` | 白送 |
| `find` | 白送 |
| `ls` | 白送 |

**Bash 权限拦截（spike 验证的实现）**：

```ts
pi.on("tool_call", async (event) => {
  if (isToolCallEventType("bash", event)) {
    const allowed = await askUserPermission(event.input.command);
    if (!allowed) return { block: true, reason: "user denied" };
  }
});
```

异步、可 await Electron IPC 弹窗。同 hook 也覆盖 read/edit/write/grep/find/ls，按需扩权限粒度。`event.input` 可 mutate，未来 SSH 远程执行不用换工具。

### Design domain 6（我们写）

| 名称 | 行为 |
|---|---|
| `scaffold(kind, path)` | 把预装 starter JSX / CSS 片段复制到 workspace（enum kind） |
| `skill(name)` | 懒加载 skill prompt 文本（manifest 在 system prompt 里列名字） |
| `ask(questions)` | 渲染结构化问卷；end turn 等用户答（5 种题型：text-options / svg-options / slider / file / freeform） |
| `preview(path)` | 渲染 artifact source（默认 `App.jsx`）→ `{screenshot?, consoleErrors, assetErrors, domOutline, metrics}`；vision model 才返 screenshot |
| `gen_image(prompt, path)` | 文生图，落盘；**capability-based 暴露**（仅 OpenAI key 配了才给 agent） |
| `tweaks(blocks)` | 声明 EDITMODE 块 + 生成 UI 控件；**跨文件**支持；全量覆盖语义 |

### Lifecycle 2

| 名称 | 行为 |
|---|---|
| `todos(items)` | 任务列表；system prompt 强制 >3 tool call 任务首调；runtime 5-call 后软提醒；UI 钉顶展示 |
| `done(path)` | 终止本轮；默认检查 `App.jsx`；有错返给 agent 修，干净则 turn 结束 |

### 已砍

- `verify` subagent（preview 自检够）
- `snip`（pi auto-compact）
- ~~独立 `list_files`、自写 bash~~（pi 自带，spike 后改主意）
- `read_design_system`（DESIGN.md 直接注入 system prompt）
- `read_url`（reference URL 由 host 预取、限流/限大小后作为 untrusted context 注入；不作为 agent 默认工具暴露）

### Capability-based 暴露规则

Session 启动时探测 `capabilities`（直接读 `Model<T>.input` / `.reasoning` / `.cost.cacheRead`，spike 验证字段齐全）：
- `gen_image` 只在 imageGen provider 配了时暴露
- `preview` 对 vision model 返截图（`Model.input.includes("image")`），对纯文本 model 返 DOM outline + metrics
- `read` 图像模式仅 vision model

## 4. Loop 结构

**Agentic，无硬相位**。

当前主线不是 tag-parser / custom streaming grammar，而是 pi function-tool loop + workspace delta stream。推荐把 harness 理解成四层：

- **Run Controller**：绑定 `designId` + `generationId`，执行 preflight ask、title/todos 约束、timeout/cancel、host phase notes。
- **Workspace Delta Bus**：把 `fs_updated`、preview pool、snapshot persist 作为按 design/generation/path 隔离的增量事件处理。
- **Context Pack / Session Memory**：brief、run preferences、workspace state、memory sections 进入 prompt；tool/artifact/error rows 不进入 LLM history。
- **Verification Gate**：`preview(path)` 提供中途检查，`done(path)` 负责最终 gate 和 repair loop。

```
SESSION_START
  ↓
  system prompt 组装（见 §5）
  ↓
TURN_START (user message + 可选 content blocks)
  ↓
AGENT_LOOP (pi-agent-core 原生)
  ├─ agent 思考 / 说话 / 调 tool（可并行）
  ├─ 工具 result 回流
  └─ 终止任一满足：
     · 调 done
     · 调 ask（等答）
     · 60 次 tool call 上限（软询问继续）
     · 10 分钟 wall clock
     · 用户 abort
  ↓
TURN_END
  ↓
  if done 被调：
    runtime 跑 preview(path)
    errors → 喂回 agent → 回 AGENT_LOOP
    clean → turn 结束，UI 显示成果
```

**软约束（写进 system prompt，不是 loop 强制）**：
- 新 / 模糊任务优先调 `ask`（10+ 问）
- 写代码前在 chat 里 declare aesthetic（palette / type / rhythm）
- 复杂任务（>3 step）首调 `todos` 规划
- 完成用 `done`，它自动 preview + 你看 console / 404 后修
- 暴露 2-5 个 tweaks（"big-rock" values，不是细枝末节）

### Multi-screen 视觉一致性（baton pattern）

借鉴 [Google Stitch](https://stitch.withgoogle.com/) 的 baton pattern——多屏 / 多 page 项目（landing + dashboard + settings 等）保证视觉一致：

```
生成屏 1
  ↓
从产物抽 token（colors / typography / spacing / radius）
  ↓
更新 DESIGN.md（agent 调 edit）
  ↓
注入下次生成 system prompt 的 [3] 层
  ↓
生成屏 2 自动用同 token system
  ↓
... 第 N 屏
```

不是新工具，是 prompt loop 设计：每生成一屏，agent 自检"DESIGN.md 是否需要更新"，更新了下一屏自然继承。配合 §5 Brand Acquisition Protocol 形成完整 token 一致性闭环。

### 中断语义（对齐 Claude Code）

```
用户 Esc / Stop：
├─ LLM streaming → AbortSignal
├─ 当前 tool call → 级联 abort
│  └─ bash：SIGTERM → 3s → SIGKILL
├─ queued user messages → 回 editor 供重编辑
└─ 后台进程（dev server）不杀，走单独 "Processes" 面板管
```

**二次 Esc（2s 内）**：弹撤销菜单（恢复到 turn 前 / 删除本 turn / 什么都不做）。

**半成品文件**：write/edit 用 atomic temp + rename，**不回滚**，中断只影响未完成的写。

### 后台进程（tab 模型）

```
design tab 打开 → background process 活
design tab 关闭 → SIGTERM 进程（关前弹确认）
切换 design tab → 什么都不做，进程继续
app 退出 → TERM → KILL 所有
```

- 每 design ≤3 进程，全局 ≤10
- Port 检测：起进程后 2s 扫 stdout 匹配 vite/next/astro/webpack 端口模式
- Processes 面板（design sidebar 固定）列活进程 + 一键 stop + 看 logs

## 5. System Prompt 层叠

```
[1] 主 system prompt                  产品级（抄 Claude Design 骨架）
    ├─ Identity（"expert designer, user as manager, filesystem-based project"）
    ├─ Workflow（软指引 7 步）
    ├─ Output 规则（filename / revisions / 避坑）
    ├─ React + Babel pinned（SRI 整段抄 Claude Design）
    ├─ Content Guidelines（反 AI slop 核心）
    ├─ Aesthetic direction（无品牌时如何承诺）
    ├─ Typography 默认（Bunny Fonts / Fontshare）
    ├─ Tools 一览（12 个每个一行）
    ├─ Tweaks EDITMODE 协议
    ├─ Ask 规则（10+ 问 + 5 种题型）
    ├─ Preview / done 协议
    └─ Do not divulge

[2] project_info                      workspace stack 探测结果（~200 字节）

[3] <project_context>                 用户项目级
      <instructions source="AGENTS.md">...</instructions>
      <design_system source="DESIGN.md">
         YAML front matter 保留原样（token 不 truncate）
         Prose 部分 truncate 到 2000 字
      </design_system>

[4] Available scaffolds (enum)         从 scaffold tool schema 的 kind 自动生成

[5] Available skills (manifest)        技法 + brand refs 两段
    ## Techniques
    - oklch_color_system: ...
    - bento_grid: ...
    ## Brand references (skill("brand:<name>"))
    linear, vercel, stripe, figma, arc, ...

[6] [对话历史]                        pi auto-compact 兜底

[7] 本轮 user message
```

**总字符目标**：base 9-10K 字符（~2.7K tokens），比 Claude Design 的 30K 小三分之二。

**DESIGN.md 生命周期**：协作式，agent 和用户都可编辑。
- Session 启动读，session prompt 包含
- Agent 第一次声明 aesthetic → 主动 `write("DESIGN.md", ...)` 生成初版
- 用户外部编辑 → FSWatcher 捕捉 → 下轮重读
- 用户持续调整某 EDITMODE 值 → agent 主动 propose "promote to DESIGN.md"

**Google DESIGN.md spec 完全采纳**（YAML front matter + markdown body，基于 W3C DTCG），不发明自己 schema。

### Brand Acquisition Protocol

灵感来自 [alchaincyf/huashu-design](https://github.com/alchaincyf/huashu-design)（A/B 测结果显示降低输出方差 5×）。写进 system prompt：

```
## Brand acquisition (never invent brand values from memory)

When user mentions a specific brand and DESIGN.md doesn't yet exist:

1. Ask the user for brand guidelines / DESIGN.md / brand site URL
2. If they give a URL: bash `curl` the /brand or /press route
3. Download SVG / HTML / screenshots as fallback (via bash + workspace storage)
4. Extract hex values programmatically from CSS / SVG, NOT from memory
5. Codify into DESIGN.md with YAML token block

NEVER write brand color hex values from memory. Always extract from
authoritative source. If the source is unavailable, ask the user.
```

理由：LLM 对 brand color 的 "记忆" 错误率高（即使 Linear / Stripe 这种知名品牌也有偏差），凭记忆生成的产物会"形似而神离"。强制走 acquisition protocol 把这个错误源头 close 掉。

## 6. 权限模型（4-Tier，抄 Claude Code）

### 分层

```
Tier 0  无感通过（不弹窗）
Tier 1  首次弹窗 + 用户选 allowlist
Tier 2  每次必弹窗（不能 suppress）
Tier 3  硬拦截（UI 拒绝，无 override）
```

### 具体归类

| 操作 | Tier |
|---|---|
| 读 workspace 内文件 | 0 |
| 写 / 改 / 删 workspace 内文件 | 0 |
| `ls` / `pwd` / `cat` / `grep` / `find` / `stat`（cwd 内） | 0 |
| 只读 git (`git status/log/diff`) | 0 |
| `pnpm install *` / `pnpm run *` / write-git | 1 |
| `curl` / `wget` 非本机 URL | 1 |
| cwd 外的任何命令 | 1 |
| `git push` / `npm publish` / `sudo` | 2 |
| `rm -rf /` / `rm -rf ~` / `curl \| sh` / 系统目录写 | **3** |

### 弹窗 UX（三按钮）

```
┌─────────────────────────────┐
│ 🟡 Agent 想运行              │
│                              │
│    pnpm install              │
│                              │
│   [拒绝]  [允许]  [总是允许]  │
└─────────────────────────────┘
```

- 默认 scope = per-design + 命令前缀（`pnpm install *`）
- 想更宽 / 更窄去 Settings 编辑 allowlist

### Allowlist 存储

`<workspace>/.codesign/settings.json`（per-design，可 commit）：
```json
{
  "schemaVersion": 1,
  "allowedTools": [
    "Bash(pnpm install *)",
    "Bash(pnpm run *)",
    "Bash(git status)"
  ]
}
```

全局：`~/.codesign/settings.json`（同格式）。加载顺序：global → project → local（settings.local.json，gitignore 默认）。

## 7. Progressive Disclosure

三层独立机制：

### 源头 budget（预防膨胀）

- `read`: view_budget（首次全文，重复摘要 + 强制 view_range，mtime 变化重置）
- `bash`: stdout tail-cap 10KB + exit code
- `preview`: screenshot 压缩 + console ≤50 条 + 404 ≤20 条
- `grep`: head_limit 250

### 按需披露

- **Skill manifest** 在 system prompt 列名字（~500 字节）
- `skill(name)` 调用时返全文（~500-2000 字节/次）
- **Scaffold enum** 在 tool schema（~200 字节）
- `scaffold(kind, path)` 执行时写磁盘 + 返短路径确认
- 一个 session 内重复 `skill(name)` → 返短"already loaded"，不重复灌

### Tool result 不做类型特定剪枝

（coding agent 都不做，我们也不）pi auto-compact 兜底。

### Tweak / Inline-edit / Comment delta

直接操纵的 delta 攒 buffer，下次 user chat 时附带到 content block：
- `<tweak_changes>`
- `<direct_edits>`（inline text edit）
- 评论立即发，不攒

| A. bash 拦截 hook | ✅ Pass | `pi.on("tool_call", async (event) => { return { block, reason } })` |
| B. Session storage | ✅ Pass | `SessionManager.create(cwd, sessionDir)` 完全扁平，文件名 `<ISO>_<uuidv7>.jsonl` |
| C. Model capability | ⚠️ Partial（实际 Pass） | `Model<T>` 有 `reasoning / input / cost / contextWindow / maxTokens` |

### v0.2 架构落地（基于 spike 结论）

- **Session**：`SessionManager.create(cwd, "<userData>/sessions")`，扁平存储
- **Bash 权限**：`tool_call` hook + 我们 Permission UI（不写新 bash 工具）
- **Capability**：`Model<T>` 直读，无 fallback 表
- **Provider**：`pi.registerProvider()` 从 config.toml 翻译注入
- **AuthStorage**：`AuthStorage.create(authPath)` 指向我们的 userData，配合 config.toml 双轨

### Spike 中发现的几个坑（写进实施 checklist）

- session 文件名是 `<ISO>_<uuidv7>.jsonl`，不是纯 uuid
- `Model.provider` 运行时是 string，不是 object（类型说谎）
- `ModelRegistry.getAll()` 不是 `getAllModels()`
- pi-ai 要 pin `^0.69.0`，不要让 pnpm 默认装 0.67.x
- 自定义 gateway 用 `pi.registerProvider("anthropic", { baseUrl })`，不靠 env

### Bonus 发现（v0.3+ 可白嫖）

- `SessionManager.forkFrom()` — 程序化 fork session
- `session.navigateTree(targetId)` — 内置版本/分支 UX
- 19+ 事件（before_provider_request / after_provider_response / compaction_start_end / auto_retry_*）
- OAuth provider 支持（未来 Claude Code OAuth 续接零代码）
- `--mode rpc` 子进程 escape hatch
