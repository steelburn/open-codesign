# v0.2 Agentic Design Loop — Design Doc

Date: 2026-04-23
Status: Draft, pending review
Target: v0.2.0 (multi-milestone)

## 0. Context

v0.1.x agent runs on `@mariozechner/pi-agent-core` with a virtual-filesystem tool (`str_replace_based_edit_tool`). Pipeline assumes single prompt → single artifact. 13 design skills are injected wholesale into the system prompt every call.

This is insufficient for the product we are becoming:

- Users increasingly want multi-page / multi-artifact projects (landing + pricing + email in one session)
- Agent cannot see user's existing files, design tokens, or previous work
- Revision flow requires re-describing the whole page
- Skill injection eats 20-40k tokens per call regardless of relevance

v0.2 upgrades the loop to a real agentic design system: read/edit a user-bound workspace, pull skills on demand, verify its own output visually, revise by pointing at regions.

## 1. Non-goals

- Not replicating Claude Code. We optimize for design-artifact iteration, not general software engineering.
- No bundled model runtime. BYOK still mandatory.
- No cloud sync of workspace content. Local-first remains a hard constraint.
- No auto-retry on LLM errors beyond what `pi-agent-core` already does. Error recovery is user-triggered ("Try to fix").

Three layers from bottom to top:

1. **Workspace layer** (Roy PR foundation + path guard): real disk, write-through, binding
2. **Harness layer** (new): permission, path sandbox, streaming parser, action runner, compaction, IPC
3. **Tool layer** (new + #177 + existing): fs + design-specific tools, progressive-disclosed skills

## 3. Mental model: Quick mode vs Project mode

Both coexist. User picks at session start; they share the tool surface but differ in persistence and loop shape.

| | Quick mode (v0.1 parity) | Project mode (new in v0.2) |
|---|---|---|
| Input | Single prompt | Long-running chat |
| Storage | Virtual fs, extracted as artifact | Real disk under bound workspace |
| Tools | `text_editor` (virtual) + image-gen (#177) + `done` | Full fs tools + image-gen + preview-capture + revision |
| Skills | Wholesale inject (current behavior) | Index-only inject, tool-pull bodies |
| Loop | 6-15 tool calls → done | Multi-turn, compacted, abort-able |
| Revision | Regenerate whole artifact | Scoped to clicked region |
| Exit | Emit artifact, close session | Session stays open, artifact evolves |

The guarantee: everything that works in v0.1 keeps working. Quick mode is the default for new users; Project mode is opt-in by binding a workspace.

## 5. Permission model

Copy opencode's `allow / deny / ask / always` rule engine. Roughly 200 lines of logic + an ordered `Ruleset` array evaluated with wildcard match (last matching rule wins).

Default ruleset for Project mode:

```ts
const DEFAULT_PROJECT_RULES: Ruleset = [
  { permission: '*', pattern: '*', action: 'ask' },
  { permission: 'read', pattern: '*', action: 'allow' },
  { permission: 'read', pattern: '*.env*', action: 'ask' },
  { permission: 'read', pattern: '.git/config', action: 'deny' },
  { permission: 'read', pattern: 'id_rsa*', action: 'deny' },
  { permission: 'external_directory', pattern: '*', action: 'ask' },
  { permission: 'image_generation', pattern: '*', action: 'ask' },
  { permission: 'edit', pattern: '*', action: 'ask' },
  { permission: 'write', pattern: '*', action: 'ask' },
];
```

UX: on the first `ask`, show a modal with `Once | Always | Deny`. `Always` appends an `allow` rule to `~/.config/open-codesign/permissions.toml`. A Permissions panel lets users review/revoke.

Reasoning for being strict by default: new users are unfamiliar; they should consent visibly to disk writes, paid API calls, and external-directory access.

## 6. Path sandbox

Builds on Roy's `normalizeWorkspacePath()`. Add a `PathGuard` wrapper that every tool goes through:

```ts
class PathGuard {
  constructor(private root: string) {}

  resolveWithin(rel: string): string {
    const joined = path.resolve(this.root, rel);
    const real = fs.realpathSync.native(joined);
    if (!real.startsWith(this.root + path.sep) && real !== this.root) {
      throw new PermissionError('external_directory', real);
    }
    return real;
  }
}
```

Also applied to Roy's existing `persistMutation` (see §10). Blocked patterns: `*.env*`, `.git/config`, `id_rsa*`, `*.pem` — matched against both source and symlink target. Matches opencode's pattern.

## 7. Phasing

Each milestone is independently shippable. Do not start milestone N+1 before N's acceptance criteria are met on main.

### M0 — v0.1.x close-out (blocker for v0.2 start)

Goal: ship current value to international users landing from the star wave.

- [ ] Roy PR #173 merged (remaining 1 Blocker + 2 Major fixed)
- [ ] Path guard PR (harden Roy's `persistMutation`, prevent `../` writes, filter `skills/`/`frames/` out of write-through)
- [ ] yangjunx21 PR #177 rebased, bot review passes, merged
- [ ] API config bug batch: #175 Gemini 400, #176 local db, #167 SSE truncation, #158, #134, #130
- [ ] Cut v0.1.x release, update CHANGELOG, Changeset

Exit criteria: fresh install works end-to-end; new user can bind a workspace and generate an HTML prototype with an AI image.

Branch policy: continue on `main`. No v0.2-dev branch yet.

### M1 — v0.2.0-alpha: Read-only agent (2 weeks)

Goal: agent can see the user's workspace without touching it yet.

- [ ] Create `v0.2-dev` long-lived branch off main
- [ ] New `projects` table + `design.project_id` FK (migration). A "project" groups designs and owns the workspace binding. Roy's per-design `workspacePath` migrates to `project.workspacePath`; per-design binding still works (single-design project).
- [ ] `PermissionManager` (§5) + `PathGuard` (§6) + IPC plumbing
- [ ] Adapt pi-coding-agent tools `read` / `ls` / `grep` / `find` as `read_file` / `list_files` / `grep` / `find`; inject `PathGuard` via `Operations` hook
- [ ] Agent system prompt: add "Explore phase" instruction (first turn reads workspace before writing)
- [ ] Quick mode remains the default; Project mode activated by binding a workspace
- [ ] Vitest coverage for path-guard escapes, ignore patterns, and permission-ask flow

Exit criteria: in Project mode with bound workspace, agent can list files, read an existing `index.html`, grep for a component name, and explain what it found — with zero write capability.

### M2 — v0.2.0-beta: Write-capable agent + skill-as-tool (2-3 weeks)

Goal: agent can edit workspace files and pull skills on demand.

- [ ] `edit_file` (str-replace) + `write_file`, wired through `PathGuard` + permission ask-once
- [ ] StreamingMessageParser on top of pi-agent-core stream (historical alternative only; not the current v0.2 mainline)
- [ ] **ActionRunner** promise chain for ordered tool execution with live UI progress
- [ ] **Snapshot-on-first-write** under `.codesign/history/`, following Bolt's pattern; expose a "Revert" button per turn
- [ ] **read_skill(name)** tool — reads from `packages/core/src/skills/builtin/` and from `{workspace}/.codesign/skills/`
- [ ] System prompt: switch from wholesale skill injection to **skill index only** (name + one-line description, ~2k tokens total for 13 skills); Quick mode retains wholesale injection
- [ ] `edit_file` preferred over `write_file` in prompt ("prefer targeted replace over full write")
- [ ] Compaction: wire up `pi-coding-agent`'s `compaction/` module; tune thresholds (20k trigger, 40k protect, last 2 turns verbatim)
- [ ] AbortSignal plumbed end-to-end; Cancel button in chat

Exit criteria: agent can open a project, read context, propose edits, write them to disk, and be safely interrupted/rolled-back. Skills are pulled on demand. Token cost per turn drops measurably (report before/after).

### M3 — v0.2.0: Visual verification + point-and-prompt revision (2 weeks)

Goal: close the generate→verify→revise loop.

- [ ] `preview_capture(viewportWidth?)` tool over Electron `webContents.capturePage()`. Returns base64 PNG for vision-capable models, plus accessibility-tree snapshot for non-vision models
- [ ] `check_token_compliance(artifactId)` reads `{workspace}/.codesign/tokens.{css,json}` (W3C DTCG or Tailwind config), returns list of literal colors/spacings that bypass tokens
- [ ] `query_artifact(cssSelector)` + `replace_artifact_fragment(selector, html)` — operates on the artifact HTML AST, not live DOM
- [ ] Renderer: click-to-select in preview iframe → compute stable CSS selector (prefer `data-testid`, then `id`, fall back to structural path) → inject as scope hint into next user prompt
- [ ] System prompt: when `scope` is present in user message, prefer `edit_file` + `replace_artifact_fragment` over regeneration
- [ ] Lightweight error-recovery UI: on tool failure, show "Try to fix" button that crafts a follow-up prompt with the error payload

Exit criteria: user generates a hero, clicks the CTA button in preview, types "make it bolder", agent modifies only the CTA (not the whole page) and re-verifies via preview capture.

### M4 — v0.2.x: Multi-artifact orchestration + assets (2+ weeks, post-GA)

Goal: one session produces a consistent suite (landing + pricing + email).

- [ ] `search_icons` — fork `marcoripa96/i0`, ship local SQLite icon index (MIT); lazy-download on first use to respect the no-bundled-runtime constraint
- [ ] `export_artifact(format)` tool wrapping `packages/exporters/` — agent can finalize deliverables
- [ ] Project-level shared-context pattern (monkilabs `project-consistency` skill): Phase 1 agent writes `project.tokens.json` + base `Layout.tsx`; Phase 2 agents import, cannot invent new tokens
- [ ] Multi-design history view in renderer; cross-design preview

Exit criteria: user says "build me a landing, pricing, and onboarding email", ends session with three consistent artifacts sharing tokens and voice, exports as ZIP.

## 8. Backwards compatibility

- Quick mode is fully preserved. Existing Vitest test `'injects every loaded skill body into the system prompt (progressive disclosure level 1)'` stays green for Quick mode.
- Per-design workspace binding (Roy's shape) continues to work. A design without a project is auto-wrapped in a "default" project at M1 migration time.
- `TextEditorFsCallbacks` interface is preserved; new fs tools go through a parallel `PathGuardedFs` interface. Both write through Roy's `persistMutation`.
- Schema migrations: every table change adds a new column or table with `schemaVersion` bumped, never drops. Roll-back only requires previous binary to work with new DB shape.

## 9. Streaming parser sketch (historical alternative, rejected for current v0.2 mainline)

Status update: the current direction is to stay on pi-agent-core function tools plus a host-side workspace delta bus. We keep this section only as a historical alternative that was considered before the current harness stabilized.

Adapt Bolt's `StreamingMessageParser` onto pi-agent-core streaming. Key idea: we feed the parser the **cumulative** stream text; it advances a position cursor and emits events on complete tags.

```ts
interface ParseState {
  position: number;
  insideArtifact: boolean;
  insideAction: boolean;
  currentArtifact?: { id: string; title: string };
  currentAction?: { id: string; type: 'file' | 'image'; filePath?: string; content: string };
}

function parse(state: ParseState, cumulativeText: string, callbacks: {
  onArtifactOpen(id: string, title: string): void;
  onActionOpen(action: { id: string; type: string }): void;
  onActionStream(id: string, contentSoFar: string): void;
  onActionClose(id: string): void;
  onArtifactClose(id: string): void;
}): ParseState { /* ~150 lines */ }
```

Our tag grammar is simpler than Bolt's — we don't need `shell` or `start`. Three action types only: `file` (write virtual or real fs), `image` (invoke `generate_image_asset`), and `note` (chat narration). System prompt will teach the model these tags.

Why it is not the chosen v0.2 mainline: it would fork the runtime protocol away from pi-agent-core's tool events, duplicate state handling, and make preview/update persistence harder to reason about under multi-design concurrency. The chosen architecture is function tools + compact tool-result persistence + per-design delta handling.

## 10. Revision UX mechanics

When user clicks in preview iframe:

1. Renderer receives click coordinates, runs `document.elementFromPoint`
2. Compute stable selector: `data-testid` → `id` → class chain (rejecting Tailwind utility chains as noise) → structural `nth-child` fallback
3. Extract element's outer HTML (to show in the chat input as a chip)
4. Next user message is sent with a `scope` metadata field: `{ selector, outerHtml, viewportWidth }`
5. System prompt instruction: "When `scope` is present, use `query_artifact` and `replace_artifact_fragment` with the given selector; do not regenerate the whole file."

## 11. Open questions

These are deliberately not answered in this doc; resolve before M3:

- Should `preview_capture` default to accessibility-tree (free) or screenshot (vision tokens) for vision-capable models? The cost/fidelity tradeoff needs an experiment.
- Revision UX: is `data-testid` a reasonable ask for generated code, or do we rely on structural selectors? Likely the former, but costs prompt tokens.
- Should `generate_image_asset` results be cached by prompt hash to prevent accidental double-billing on retries? Probably yes; tracks memory feedback on silent fallback concerns.
- For compaction, do we keep `pi-coding-agent`'s default `summary` prompt, or author a design-specific one ("Keep decisions about color, typography, and voice; drop file-level diffs")? Worth an A/B.
- Does Figma MCP plug in as an optional data source at M4, or wait for v0.3? Many users don't have Figma.

## 12. Risks

- **pi-agent-core loop hold-up**: our M2 introduces a custom streaming parser on top of pi-agent-core's tool-call stream. If pi-agent-core's internals change between minor versions, we break. Mitigation: pin minor version, maintain adapter layer isolated to one file.
- **Token cost blow-up**: multi-turn Project mode with a 50-file workspace can easily eat 100k tokens/turn if compaction misbehaves. Mitigation: telemetry-less (local only) token counters shown to user; hard cap with user-override.
- **Permission fatigue**: too many `ask` dialogs will annoy users. Mitigation: `Always` action persisted per-project; the first 5 operations of a session can carry a "trust for this session" bulk-approve.
- **Skill miss**: Tier-1 index might fail to trigger the right skill lookup (Anthropic docs explicitly warn about this). Mitigation: write precise one-line descriptions; include representative task examples in skill frontmatter; ship a test suite that asserts the model picks the correct skill on sample prompts.
- **Scope creep into coding-agent territory**: users will ask for `bash`, `git`, `npm`. We say no in v0.2. Mitigation: clear docs on what Project mode is and is not. The differentiation is "agent for design artifacts", not "VS Code replacement".

## 13. Success metrics

Measured locally (no telemetry). Report in the v0.2 release notes.

- Median tokens per turn drops ≥30% (skill progressive disclosure)
- Median revision task completes ≥2× faster (scope hint + targeted edit)
- Agent retrieves correct skill on ≥85% of sample prompts (offline test set)
- No P0 regressions in Quick mode per existing Vitest suites
